# dstack 私有化部署向导(GCP)

面向**客户自有 GCP 环境**的端到端部署。全文用 shell 变量占位,不绑定任何具体项目,
按需替换即可。四个部分:制作加密业务镜像 → 部署 Authority → 部署 KMS → 部署业务 CVM,
最后是出网白名单硬化。

---

## 总览:三个角色与信任模型

```
┌─────────────┐   厂商掌控授权(per-user 根密钥 / 全局镜像密钥环 / 白名单)
│  Authority   │   厂商侧主机上跑 docker-compose(authority + dstack-verifier)
│ (厂商主机)   │   离线签发 AuthBundle、HPKE 封装根密钥、校验 KMS 远程证明
└──────┬──────┘
       │ courier(CLI 中继,经 IAP 隧道)
┌──────┴──────┐   GCP TDX+vTPM 机密虚机
│   KMS CVM   │   dstack-kms + key-broker:拿到根→派生各 app 密钥、租出镜像私钥环
└──────┬──────┘
       │ mTLS(VPC 内网)
┌──────┴──────┐   GCP TDX+vTPM 机密虚机
│  业务 CVM    │   launcher:取镜像私钥→JWE 解密 ocicrypt 加密镜像→运行业务容器
└─────────────┘
```

**原理**:业务镜像被**层加密**(ocicrypt 原生 JWE,非对称——只用公钥加密),解密所需的
**私钥**只在通过远程证明后、由 KMS 在 TEE 内租下发。厂商通过 Authority 控制"哪台机器、跑哪个
镜像"才能拿到私钥——客户的 GCP 管理员、GCP 本身都拿不到明文镜像和根密钥。镜像私钥是**全局**
的(一份加密镜像发所有租户);各租户的 app/disk 密钥则由**各自独立的根**派生,跨租户隔离。

## 0. 准备

```bash
# —— 按客户环境替换 ——
export PROJECT=<your-gcp-project>
export REGION=us-central1
export ZONE=${REGION}-a
export AR_REPO=dstack-private                       # Artifact Registry 仓库
export AR=${REGION}-docker.pkg.dev/${PROJECT}/${AR_REPO}
export BUCKET=gs://<your-dstack-bucket>             # dstack-cloud 部署用的桶
export USER_ID=acme-prod                            # 租户 ID(多租户时每客户一个)
export AUTHORITY=http://localhost:8083               # Authority 监听地址(厂商主机)
export SSH_KEY=$HOME/.ssh/id_ed25519                # 已登记到 dstack OS 的密钥
```

**目的**:CVM 无公网,镜像必须放在客户 GCP 内部 registry,经 **Private Google Access**
拉取;Authority 跑在厂商能访问 KMS 的主机上。

```bash
# 启用 API
gcloud services enable compute.googleapis.com artifactregistry.googleapis.com \
  networksecurity.googleapis.com networkservices.googleapis.com --project=$PROJECT
# 建 Artifact Registry 仓库 + 部署桶
gcloud artifacts repositories create $AR_REPO --repository-format=docker \
  --location=$REGION --project=$PROJECT
gcloud storage buckets create $BUCKET --project=$PROJECT --location=$REGION
# 工具:dstack-cloud CLI、skopeo、openssl、docker、gcloud
```

> **key-broker / launcher / dstack-verifier** 从本仓库 `build`(各自的 Dockerfile);
> **dstack-kms** 用官方/预构建镜像(`dstacktee/dstack-kms`——mainline 的 ra-tls 已带 IP SAN,
> 无需自 build)。都用 `scripts/sync-image.sh` 同步进 `$AR`。本向导假设它们已在 `$AR` 中。

## 1. 制作加密业务镜像

**目的**:把业务镜像加密成"没有对应私钥就跑不起来";**加密只用公钥**,私钥由 Authority 托管、
经 AuthBundle 租给 TEE 内的 launcher。
**原理**:ocicrypt **原生 JWE**(EC P-256 / ECDH-ES)。`skopeo --encryption-key jwe:<pub.pem>`
用公钥把每层的层密钥包好;launcher 解密时把租来的私钥喂给 `skopeo --decryption-key <priv.pem>`,
ocicrypt 自己匹配 recipient。**无自定义 keyprovider**——skopeo/ocicrypt 内建 JWE。

> **前置**:厂商**全局镜像公钥** `pub.pem`。它由 §2 的 `POST /api/v1/admin/keys` **mint 一次**、
> 长期复用(一把密钥用一段时间、跨多次构建与所有租户);轮换时 mint 新 kid 再用新公钥加密即可。
> 公钥不敏感,可放进 CI;**构建机只有公钥,泄露也解不了任何镜像**。

```bash
# 0) 取厂商全局镜像公钥(一次性;见 §2 mint 后保存复用)
#      curl POST /api/v1/admin/keys → .pub_pem  存成 pub.pem

# 1) 用公钥加密并推送到「公共 registry」(加密后层无明文,可放公共仓库)
#    构建/加密在能上公网的 CI/开发机完成;CVM 无公网,下一步再同步进客户 GCP 私有 AR。
export PUB_REG=docker://<your-public-registry>/<your-app>-enc:latest   # 如 ghcr.io / docker.io / cr.kvin.wang
skopeo copy --encryption-key jwe:pub.pem \
  docker://<your-app:tag> \
  "$PUB_REG"

# 2) 用 sync-image.sh 把加密镜像同步进 GCP 私有 AR(skopeo copy --all 按摘要逐层搬,
#    加密层原样复制、摘要不变;CVM 经 Private Google Access 从 AR 拉)
#    config.env 需含 AR_LOCATION/AR_PROJECT/AR_REPO;脚本末行打印 <AR-ref>@<digest>
DEST_REF=$(scripts/sync-image.sh "${PUB_REG#docker://}" "<your-app>-enc:latest" | tail -1)
export IMAGE_DIGEST="${DEST_REF#*@}"     # sha256:...
echo "ar-ref=$DEST_REF  digest=$IMAGE_DIGEST"
```

> **要点**:launcher 按 `@<digest>` 从私有 AR 拉取(摘要=版本指针+完整性);`skopeo copy --all`
> 保留摘要,公共 registry 与 AR 中一致。**私钥不进镜像、不进 compose**,只在 TEE 内经租约下发;
> 加密只需公钥。同一把全局密钥跨多次构建/多个租户复用,**无需每镜像注册 CEK**。

摘要会在第 2 步作为"当前版本指针"注册到 Authority(不再注册 CEK)。

## 2. 部署 Authority(厂商侧)

**目的**:厂商的授权中枢。它持久化**每租户独立的根密钥**(per-user,派生各租户专属密钥)、托管
**全局镜像密钥环**(vendor-wide,所有租户共用同一把解密私钥)、签发 Ed25519 的 **AuthBundle**
(谁能跑、跑哪个摘要、附全局私钥 keyring)、并用 **dstack-verifier** 校验 KMS 的 TDX+vTPM 证明。
**原理**:Authority + verifier 用 docker-compose 跑在厂商主机;源码 `./authority` 挂载进容器,
改完 `docker restart` 即生效。签名私钥落盘持久化 → 平台公钥稳定。镜像 keyring 存在独立的
`keyring.json`(全局),根密钥存在 `users.json`(per-user)。

```bash
# .env.authority(厂商密钥,勿入库)
cat > .env.authority <<EOF
AUTHORITY_SIGNING_KEY=$(openssl rand -hex 32)        # Ed25519 种子,持久化→公钥稳定
AUTHORITY_NONCE_SECRET=$(openssl rand -hex 32)       # 无状态 challenge 的 HMAC 密钥
AUTHORITY_ADMIN_TOKEN=$(openssl rand -hex 16)        # 管理接口 Bearer token
REQUIRE_ATTESTATION=true
EXPECTED_OS_IMAGE_HASH=                             # 见 §3.4"获取 OS 镜像哈希"(measure 命令)
KMS_DOMAIN=10.128.15.220                            # KMS rpc 证书的 SAN,必须 == kms_urls 的主机(静态内网 IP)
ALLOWED_TCB_STATUSES=UpToDate,SWHardeningNeeded
EOF
export ADMIN_TOKEN=$(grep AUTHORITY_ADMIN_TOKEN .env.authority | cut -d= -f2)

# 用 --env-file 让容器拿到这些 env(改了 env 要 up -d 重建,docker restart 不会重载 env)
docker compose --env-file .env.authority -f docker-compose.authority.yml up -d --build
curl -s $AUTHORITY/api/v1/authority-pubkey            # → {"pubkey":"..."} 记下,KMS 要用
```

**(a) mint 全局镜像密钥**(一次性;返回的公钥给 §1 加密用,私钥永不出 API):

```bash
# 全局密钥环(vendor-wide):mint 一把 EC P-256,长期复用、跨所有租户
curl -s -X POST $AUTHORITY/api/v1/admin/keys \
  -H "Authorization: Bearer $ADMIN_TOKEN" -H 'Content-Type: application/json' \
  -d '{"kid":"img-2026q2"}' | python3 -c 'import sys,json; print(json.load(sys.stdin)["pub_pem"])' > pub.pem
cat pub.pem        # → -----BEGIN PUBLIC KEY-----  这就是 §1 的 jwe:pub.pem
# 轮换:再 mint 新 kid(如 img-2026q3),旧 kid 仍在 keyring 直到对应镜像下线;撤销:
#   curl -X DELETE $AUTHORITY/api/v1/admin/keys/<kid> -H "Authorization: Bearer $ADMIN_TOKEN"
```

**(b) 建租户 + 注册业务 app**(摘要作为当前版本指针;**不再注册 CEK**):

```bash
# 建租户(返回该租户 API key,多租户时分发给客户操作员)
curl -s -X POST $AUTHORITY/api/v1/admin/users \
  -H "Authorization: Bearer $ADMIN_TOKEN" -H 'Content-Type: application/json' \
  -d "{\"user_id\":\"$USER_ID\"}"

# 注册业务 app + 当前加密镜像摘要(版本指针;解密私钥来自全局 keyring,不在此注册)
curl -s -X POST $AUTHORITY/api/v1/admin/users/$USER_ID/images \
  -H "Authorization: Bearer $ADMIN_TOKEN" -H 'Content-Type: application/json' \
  -d "{\"app_id\":\"<workload-app-id>\",
       \"allowed_launcher_digests\":[\"*\"],
       \"image_digest\":\"$IMAGE_DIGEST\"}"
```

> `app_id` 是业务 CVM 的应用 ID(见第 4 步 app.json)。`allowed_launcher_digests` 是允许的
> launcher compose 哈希集合(`*` 仅用于联调,生产应填具体哈希)。注册后,下一次 KMS
> provision/同步会把它写进 AuthBundle 的 `app_whitelist`;**全局 keyring 的私钥**也随每个
> 租户的 AuthBundle 下发——所以用全局公钥加密的镜像,任一获授权的业务 CVM 都能解。

## 3. 部署 KMS CVM

**目的**:在 GCP TDX 机密虚机里跑 KMS,从 Authority 安全拿到根密钥后,对外提供密钥服务、
给业务 CVM 下发镜像私钥环租约。
**原理**:KMS 自举不能依赖 `.env`(dstack 的 `.env` 要用 KMS 派生的密钥解密 → 循环),所以
**`key_provider=tpm`**(GCP vTPM 封装磁盘),平台公钥以**字面量**写进 compose(从而进入被度量的
`compose_hash`,可被证明防篡改)。

**3.1 配置** `deploy/kms-prod/app.json`(关键字段):

```jsonc
{
  "name": "kms-prod",
  "gcp_config": {
    "project": "<PROJECT>", "zone": "<ZONE>",
    "instance_name": "dstack-kms-prod",
    "machine_type": "c3-standard-4",          // c3-* = 支持 TDX
    "bucket": "gs://<your-dstack-bucket>",
    "private_ip": "10.128.0.40"               // 固定内网 IP(见下),业务 CVM 要按它连
  },
  "key_provider": "tpm",                       // GCP vTPM,非 kms
  "secure_time": false                         // 关闭授时依赖,免公网 NTP 阻塞启动
}
```

```bash
# 固定内网 IP:先预留为静态(remove/deploy 后地址不变,业务 CVM 的 KMS_URL 才稳定)
gcloud compute addresses create dstack-kms-prod-ip --region=$REGION \
  --subnet=default --addresses=10.128.0.40 --project=$PROJECT
```

**3.2 配置 compose**(`deploy/kms-prod/docker-compose.yaml`)——两个服务:
- `key-broker`:挂 `/var/run/dstack.sock`(向导出 TDX/vTPM 证明),环境变量
  `AUTHORITY_PUBKEY=<第2步公钥字面量>`(启用 AuthBundle 验签)。
- `kms`:`depends_on` key-broker 健康;内联 `kms.toml`(`pccs_url=https://api.trustedservices.intel.com`,
  直连 Intel PCS,无 PCCS 虚机);若做出网白名单,加 `HTTP_PROXY/HTTPS_PROXY=http://<SWP_IP>:80`(见第 5 节)。

**3.3 部署 + 开放 courier 端口**:

```bash
DC=dstack-cloud
$DC -C deploy/kms-prod prepare        # 生成 shared/(app-compose、sys-config)
$DC -C deploy/kms-prod deploy         # 创建 TDX 虚机(用上 private_ip)
$DC -C deploy/kms-prod fw allow 8001 8002   # 放行 IAP → key-broker(courier/mTLS)
```

**3.4 provision(courier 远程证明)** —— **CLI 充当 courier**,在厂商主机上中继 Authority 与
in-VPC 的 key-broker,所以即使 KMS 无公网也能完成:

```bash
# 获取 OS 镜像哈希(首次):用只读 measure 命令(只验 quote、打印测量值,不释放根),
# 把打印的 os_image_hash 填进 .env.authority 的 EXPECTED_OS_IMAGE_HASH 并重建平台;同时写一份
# auth_hash.txt 让 CVM 自身也 pin 上(否则 BootInfo.os_image_hash 为空,被 fail-closed 拒):
#   echo <os_image_hash> > ~/.dstack/images/<os_image>/auth_hash.txt
gcloud compute start-iap-tunnel dstack-kms-prod 8001 \
  --local-host-port=localhost:8001 --project=$PROJECT --zone=$ZONE &
python3 authority/kms_ctl.py measure \
  --user-id $USER_ID --kms-url http://localhost:8001 --authority-url $AUTHORITY   # 打印 os_image_hash 等
python3 authority/kms_ctl.py attest \
  --user-id $USER_ID --kms-url http://localhost:8001 --authority-url $AUTHORITY
```

四步流程:① challenge(平台发 HMAC nonce)→ ② courier/init(key-broker 出 TDX+vTPM 证明,
`report_data=SHA512(nonce‖transport_pub‖kms_ts)` 绑定本次传输密钥)→ ③ provision(平台经
verifier 校验,过 **KMS 身份白名单**后 HPKE 封装根密钥 + 签 AuthBundle)→ ④ courier/install
(key-broker 验签、HPKE 解封、落地 keyset)。

> **KMS 身份白名单**(平台释放根密钥前的三项稳定校验,刻意**不用** `mr_aggregated`——GCP 的
> PCR0 每实例都变):①`os_image_hash`==`EXPECTED_OS_IMAGE_HASH`;②`key_provider`==`tpm`;
> ③`compose_hash` ∈ 白名单(显式 `allowed_kms_compose_hashes` 或首次信任 TOFU)。
> 这三项跨重部署稳定,所以 KMS 重部署**不需要**清任何 pin。

**3.5 验证**:

```bash
gcloud compute ssh root@dstack-kms-prod --tunnel-through-iap --zone=$ZONE \
  --ssh-flag="-i $SSH_KEY" --command='curl -sk https://localhost:8000/prpc/KMS.GetMeta | head -c 60'
# → {"ca_cert":"-----BEGIN CERTIFICATE..."}  即 KMS 已从平台下发的根启动并提供 TLS
```

## 4. 部署业务 CVM

**目的**:运行 launcher,在 TEE 内取得镜像解密私钥、解密业务镜像并运行,全程无公网。
**原理**:业务 CVM 用 **`key_provider=kms`** 连我们的 KMS——guest-agent 启动时就和 KMS 做
RA-TLS 拿到 app 密钥 + KMS 派生 CA(KMS 经 `/bootAuth/app` webhook 按 AuthBundle 的
`app_id`/`compose_hash`/`os_image` 白名单授权它启动)。launcher 调 guest-agent 的
`get_tls_key(usage_ra_tls=true)` 拿到**KMS 签发、带 app_info 扩展的 RA-TLS 客户端证书**,
据此连 key-broker(mTLS)申请 **密钥租约**(key-broker 用 `WebPkiClientVerifier(KMS CA)` 验链 +
从扩展取 `app_id`/`compose_hash`/`os_image` 校验白名单后,返回**全局镜像私钥环**),launcher 把私钥
写入 tmpfs 并全部喂给 `skopeo --decryption-key` 按摘要解密(ocicrypt JWE 自动匹配 recipient)、
`docker load`、compose 起容器。租约有 TTL,断租超宽限期停业务容器(反持久化滥用)。

**4.1 配置** `deploy/launcher-prod/app.json`:**`key_provider=kms`**、`secure_time=false`、
`gcp_config.private_ip=10.128.0.41`(预留静态内网 IP)、`app_id=<workload-app-id>`(与第 2 步注册一致)。
并在 dstack-cloud 全局配置(`dstack-cloud config-edit`)把 `services.kms_urls` 指向我们的 KMS
(`https://<KMS 内网 IP>:8000`),否则默认会连公网 `kms.tdxlab.dstack.org`。

**4.2 配置 compose**(`deploy/launcher-prod/docker-compose.yaml`)——launcher 服务的环境变量:

```yaml
environment:
  - SIDECAR_URL=https://10.128.0.40:8002    # key-broker mTLS(取镜像私钥租约)
  - APP_ID=<workload-app-id>
  - WORKLOAD_IMAGE=<AR>/<your-app>-enc      # 不含 tag;launcher 按 version 给的摘要拉取
  - LEASE_TTL=3600
volumes:
  - /var/run/docker.sock:/var/run/docker.sock   # 在宿主机上管理业务容器
  - /var/run/dstack.sock:/var/run/dstack.sock   # 调 guest-agent get_tls_key(KMS 签发 RA-TLS 证书)
```

**4.3 部署 + 验证 E2E**:

```bash
$DC -C deploy/launcher-prod prepare && $DC -C deploy/launcher-prod deploy
# launcher 启动后自动跑通:取私钥租约 → JWE 解密 → 运行。查看:
gcloud compute ssh root@dstack-launcher-prod --tunnel-through-iap --zone=$ZONE \
  --ssh-flag="-i $SSH_KEY" --command='docker ps; docker logs dstack-launcher-1 2>&1 | tail'
# 期望:lease acquired → decrypting docker://...-enc@sha256:... → workload containers started
```

## 5. 出网域名白名单硬化(推荐)

**目的**:让两台 CVM 只能访问被批准的目的地;业务 CVM 彻底无公网,KMS 仅能到 Intel PCS。
**原理**:三层,用网络标签 `dstack-cvm` 限定只作用于这两台,不影响 VPC 内其它机器。

**① Private Google Access**:子网开 PGA;每台 CVM 用 `/etc/hosts` 把 `*.googleapis.com`/
`*.pkg.dev` 钉到私有 VIP `199.36.153.10`(不动 VPC DNS)。AR/GCS 走 Google 私网,无公网。

**② 业务 CVM 全锁**:打 `dstack-cvm` 标签 + 摘除外网 IP + egress-deny 防火墙(只放行内网/
PGA VIP `:443`/SWP/metadata)。AR 经 PGA、KMS 经内网,够用。

**③ KMS 经明文 SWP 出 Intel PCS**:KMS 的 dcap-qvl 用 `reqwest+rustls`,**不信任**自签的 HTTPS
代理证书且无 env 可加 CA;故把 GCP **Secure Web Proxy 建成明文端点**(`ports:[80]`,不配证书)。
rustls 用明文 `CONNECT` 隧道、对 Intel 端到端 TLS,SWP 按 SNI/CONNECT 主机名做白名单。
白名单要放 **两个** Intel 域名:`api.trustedservices.intel.com`(TCB/QE identity)**和**
`certificates.trustedservices.intel.com`(验 quote collateral 取 IntelSGXRootCA.der)——少一个
KMS 验业务 CVM 的 quote 时就会 `tunnel error: unsuccessful`。KMS compose 设 `HTTP_PROXY=http://<SWP_IP>:80`。

> 一键脚本见 `scripts/setup-swp.sh`(pga / gateway / hosts / lockdown / verify 各阶段)。
> 验证:CVM 内 `curl -x http://<SWP_IP>:80 https://api.trustedservices.intel.com/...` = 200,
> 直连 `https://www.google.com` 超时。

## 6. 接口与安全(生产无 SSH)

**目的**:生产环境**不安装 sshd**;CLI 与 KMS/业务的一切交互都走专门设计的 HTTP API,
且严防经 API 泄漏 CVM 内敏感数据。
**原理**:SSH 仅开发期用(prelaunch 读实例 metadata `dev-ssh=1` 才装,`DEPLOY_MODE=prod`
默认不带 → 零 sshd)。运维/部署所需操作都已有 HTTP 端点。

**HTTP 接口面**:

| 端点 | 端口/传输 | 用途 |
|------|-----------|------|
| `/courier/init`、`/courier/install` | 8001,IAP | provision(出证明、装 HPKE 封装的根 + 验签 AuthBundle) |
| `/bootAuth/kms`、`/bootAuth/app` | 8001,本地 | **KMS 的授权 webhook**:KMS 启动时校验自身、各 app 取密钥时校验其 app_id/compose_hash 是否在白名单 |
| `/healthz`、`/version`、`/usage-receipt` | 8001 | 就绪(root 未到返回 503)、当前镜像摘要+bundle_seq、平台签名用量回执 |
| `/lease/acquire`、`/lease/renew` | 8002,**mTLS** | 给业务 CVM 下发镜像私钥环 + 租约 |
| launcher `/status`、`/healthz` | 9100 | 只读运行状态(摘要/租约/seq/是否运行/错误类别),替代 `docker logs/ps` |

**KMS 自动启动**:provision(纯 HTTP courier)写下 `/kms/_ready` 后,KMS 容器的等待循环
自动 `exec dstack-kms`——**无需 SSH 重启**。

**API 安全模型(防泄漏)**:
1. 构造上不外泄密钥:端点只回 HPKE 封装/加密 blob、签名/公钥证书、布尔 + 非敏感元数据。
   根密钥、镜像私钥环——任何**明文管理端点**都不返回(mint 只回公钥)。
2. **唯一私钥出口 = 镜像私钥环经 mTLS** `/lease/acquire`,且绑定已远证 + 已授权(app_id ∈
   app_whitelist、compose_hash/os_image ∈ 白名单)的 launcher,给最小必要。
3. 无内省/调试/文件/exec 端点;`/status` 不回日志、不回敏感数据。
4. 每端点强认证:install 只认平台 Ed25519 签名的 AuthBundle + `bundle_seq` 单调防回滚;
   `/lease/*` 用 mTLS;`/healthz`、`/version` 等只读非敏感。
5. 传输与可达性:全程 TLS(courier 仅经 IAP,无公网 IP);CVM egress-deny,即便泄漏也无外发路径。
6. 错误响应可含文件路径(便于排障),但**不含密钥内容**。

**Fail-closed 原则**:所有授权/校验逻辑**默认拒绝**,空白名单/缺失配置/缺证明一律拒;
放宽必须是**显式、会打日志的开关**,绝不默认放行。强制门(均 fail-closed):
- `os_images` 白名单(bootAuth + 私钥租约都查;空=拒)——平台按 `EXPECTED_OS_IMAGE_HASH` 填。
- `key_provider==tpm`、`compose_hash` ∈ 白名单(KMS provision 时,平台侧)。
- 私钥租约要求 launcher 出**已远证的 RA-TLS 证书**(取 app_id/compose_hash/os_image);无证则拒;
  空 keyring 也拒(fail-closed)。
- TCB 状态 ∈ `allowed_tcb_statuses`(空/未知=拒);device 绑定空列表=拒(GCP 显式 `allow_any_device`)。

**无任何"关安全检查"的开关**:为减小攻击面与审计负担,所有 insecure 旁路已**删除**
(原 `ALLOW_INSECURE_LEASE`/`ALLOW_INSECURE_LAUNCHER`/`ALLOW_DEV_PLACEHOLDER`/
`ALLOW_OS_IMAGE_DISCOVERY` 及 bundle 的 `allow_any_os_image`/`allow_any_tcb` 全部去掉)。
代码只有 fail-closed 一条路径。

- **首次发现 OS 镜像哈希**:用只读命令 `kms_ctl.py measure`——只验证 quote 真伪并打印
  `os_image_hash`/`compose_hash`/`key_provider`,**绝不释放根密钥**;据此设 `EXPECTED_OS_IMAGE_HASH`。
- `allow_any_device` 保留为 bundle 里的**显式正向策略字段**(GCP `device_id` 不 pin 的有意声明),
  不是 env 开关。

> 注:改动涉及 key-broker(Rust)+ 平台逻辑,**需重 build+sync key-broker 镜像 + 下次 provision**
> 才生效。生效后业务 CVM 必须出**真 RA-TLS**(内嵌 TDX+vTPM quote)才能拿镜像私钥——见下"业务 CVM
> 身份认证"。

## 附:常见问题

- **provision 报 403 `not in whitelist`**:通常是平台跑着旧 `mr_aggregated` 机制(GCP PCR0 每实例变)。
  现机制 key 在 `os_image_hash+key_provider=tpm+compose_hash`,重部署不需清 pin;若见到,
  重启平台载入新代码。**故意改 KMS compose** 会合法 403,需把新 `compose_hash` 加进
  `allowed_kms_compose_hashes`。
- **内网 IP 重部署后变了**:用 `gcp_config.private_ip` + 预留静态地址(第 3 步)。
- **AR 拉取 `Unauthenticated`**:CVM 的 prelaunch 要用实例 SA 的 metadata token `docker login`
  到 AR(dstack OS 的 python 无 `json`,用 `sed` 解析 token)。
- **KMS 卡在 onboarding 页**:`kms.toml` 没下发,KMS 用了内置默认值;必须用 compose `configs:`
  内联并以 `dstack-kms -c /etc/dstack/kms.toml` 启动。
- **skopeo JWE 解密报 `missing private key` / docker API 太旧**:私钥须是 **PKCS#8**
  (`-----BEGIN PRIVATE KEY-----`;SEC1 的 `EC PRIVATE KEY` ocicrypt 不收——平台 mint 已用 PKCS#8);
  用 ubuntu 24.04 的 skopeo 1.13+,并解密到 `docker-archive` 再 `docker load`(绕开 skopeo 固定
  的旧 docker API)。注意解密 `--decryption-key <priv.pem>` **不带** `jwe:` 前缀(前缀只用于加密)。

### 业务 CVM(key_provider=kms)联调踩坑(实测验证过)

无 SSH 时怎么看 CVM 内日志:`public_logs:true` 下经 guest-agent **8090** 端口拉容器日志
`GET /logs/<container>?text&tail=N`,启动状态看 launcher **9100** 的 `/status`——都走 IAP 隧道,不用 SSH。

dstack-prepare 启动循环(boot loop)= 业务 CVM 拿不到 KMS 密钥,从 8090 日志看 dstack-prepare 报错:

- **`certificate not valid for name "<IP>"; only valid for DnsName("kms.local")`**:`kms_urls` 用 IP 但
  KMS 证书没 IP SAN。要点:① ra-tls 用 `CertificateParams::new(alt_names)`(rcgen 自动出 IP SAN);
  ② `KMS_DOMAIN`/`kms_urls` 一致(用静态内网 IP);③ **dstack-kms 镜像的 ra-tls 要支持 IP SAN**(KMS
  启动会 `update_certs` 重生成 8000 证书)——基于 mainline 或官方 `dstacktee/dstack-kms` 即可,旧 fork
  镜像缺该修复才需自行重 build。
- **`invalid quote: Failed to get collateral: ...certificates.trustedservices.intel.com...: tunnel error`**:
  SWP 白名单漏了 `certificates.trustedservices.intel.com`(见 §5,两个 Intel 域名都要放)。
- **`Boot denied: OS image is not allowed`**:业务 CVM 的 `os_image_hash` 为空(没 pin)。建
  `~/.dstack/images/<os_image>/auth_hash.txt`=EXPECTED_OS_IMAGE_HASH,prepare 会写进 vm_config。

launcher 容器崩溃重启(boot 过了但 `/status` 连不上),从 8090 看 launcher 日志:

- **`GetTlsKey returned 404` / `400 Service not found`**:调 guest-agent 的 prpc 路径错。dstack.sock 上
  `DstackGuest` 挂在 `/` 且无 trim,正确路径是裸方法名 **`/GetTlsKey`**(不是 `/prpc/...` 也不是
  `/DstackGuest.GetTlsKey`)。
- **mTLS 连 key-broker 失败**:key-broker 8002 必须发 **KMS-CA 签发的 rpc 证书**(带 IP SAN)且
  **强制**验客户端证书(不能 self-signed / no_client_auth);launcher 用 GetMeta 的 KMS CA 验服务端、
  用 guest-agent `get_tls_key` 的 KMS 签发证书做客户端身份。

> 完整实测链路:KMS(fail-closed + IP SAN cert + 经明文 SWP 出 Intel)→ 业务 CVM `key_provider=kms`
> 启动取钥解盘 → launcher `get_tls_key` 拿 KMS 签发 RA-TLS 证书 → key-broker 双向 mTLS → 租约 +
> **全局镜像私钥环** → skopeo `--decryption-key` JWE 解密 → 业务容器运行。零 insecure 开关,全程无 SSH。
